# Спецификация: Сайт формирования ресурсного плана

## 1. Назначение

Система предназначена для пошагового формирования ресурсного плана на основании:

- Excel-файла графика отпусков;
- сохраненного состава команд;
- настроек ролей, видов работ и распределения времени;
- производственного календаря;
- Excel-шаблона итогового файла.

Результат работы системы:

- анализ выбранного состава по ролям, должностям, спринтам и рабочим дням;
- формирование и скачивание Excel-файла ресурсного плана;
- сохранение и повторное использование состава команд и настроек.

## 2. Технологический стек

### 2.1. Backend

- Node.js;
- Express 5;
- CommonJS-модули;
- Multer для загрузки Excel-файлов.

### 2.2. Frontend

- HTML;
- CSS;
- Vanilla JavaScript без frontend-фреймворка.

### 2.3. Работа с Excel

- ExcelJS;
- xlsx-populate.

### 2.4. Хранение данных

- JSON-файлы;
- база данных не используется.

### 2.5. Тестирование

- встроенный `node:test`.

## 3. Структура проекта

Основные файлы и каталоги:

- `src/server.js` — HTTP-сервер и API;
- `src/services/bootstrap.js` — инициализация, парсинг исходных данных и дефолтных настроек;
- `src/services/calendar.js` — работа с календарным API и fallback по рабочим дням;
- `src/services/plan.js` — расчет ресурсного плана;
- `src/services/export.js` — генерация итогового Excel-файла;
- `src/services/storage.js` — файловое чтение и запись состояния;
- `src/utils/date.js` — вспомогательная логика дат;
- `public/index.html` — разметка интерфейса;
- `public/styles.css` — стили интерфейса;
- `public/app.js` — клиентская логика интерфейса;
- `test/*.test.js` — автотесты.

## 4. Хранилища

Система использует следующие файлы:

- `data/settings.json` — настройки приложения, команд, ролей, видов работ и состава;
- `data/vacations.json` — импортированный график отпусков в JSON;
- `Ресурсный план.xlsx` — шаблон итогового Excel-файла.

## 5. Источники данных

### 5.1. График отпусков

Пользователь загружает Excel-файл в исходном формате. Перед ручной подготовкой файл не требует промежуточного преобразования.

Из файла должны считываться:

- подразделение;
- ФИО сотрудника;
- должность;
- периоды отпусков;
- суммарное количество отпускных дней.

После загрузки файл преобразуется в JSON и используется на следующих этапах.

### 5.2. Производственный календарь

Рабочие дни рассчитываются на основании внешнего API производственного календаря.

Настраиваемые параметры API:

- использование API;
- системное имя ресурса;
- отображаемое название;
- базовый URL;
- путь endpoint;
- HTTP-метод;
- имя параметра даты начала;
- имя параметра даты окончания;
- имя параметра разделителя;
- значение разделителя;
- значение рабочего дня;
- тип авторизации;
- имя параметра API-ключа;
- API-ключ;
- код страны;
- таймаут;
- режим fallback;
- примечание.

Если API недоступно или возвращает некорректные данные, применяется fallback-логика по будням.

## 6. Общая архитектура

Приложение реализовано как монолитное веб-решение:

- сервер на Express отдает статические frontend-файлы;
- frontend работает через вызовы REST API;
- вычислительная логика вынесена в сервисы;
- состояние хранится на файловой системе в JSON;
- итоговый Excel формируется на сервере по шаблону.

## 7. Основные пользовательские сценарии

### 7.1. Формирование нового плана

Пользователь:

1. загружает файл графика отпусков;
2. выбирает команду;
3. выбирает подразделения;
4. выбирает сотрудников;
5. выбирает период;
6. запускает анализ;
7. формирует и скачивает Excel-файл.

### 7.2. Повторное использование команды

Если у команды уже есть сохраненный состав, система может:

- подставить его при следующей работе;
- сопоставить с новым файлом отпусков;
- показать предупреждение о не найденных данных;
- обновить состав по текущему файлу;
- полностью очистить состав команды.

## 8. Интерфейс главной страницы

### 8.1. Общая компоновка

Главная страница состоит из:

- верхней панели с кнопкой открытия настроек;
- блока-заголовка с текстом `Ресурсный план за несколько шагов`;
- центральной рабочей области с аккордеоном шагов;
- правого степпера;
- нижней панели статуса и кнопки `Сброс`.

### 8.2. Этапы мастера

На главной странице доступны этапы:

1. Загрузка файла
2. Сохранение данных
3. Подразделения
4. Сотрудники
5. Период
6. Анализ
7. Ресурсный план

Каждый этап оформлен как аккордеон со статусом выполнения.

## 9. Логика шагов

### 9.1. Шаг 1. Загрузка файла

Пользователь загружает Excel-файл отпусков.

После успешной загрузки:

- данные преобразуются в JSON;
- обновляются подразделения и сотрудники;
- активируются следующие шаги мастера.

### 9.2. Шаг 2. Сохранение данных

Показывается список команд.

Требования:

- выбирается только одна команда;
- при наличии сохраненного состава система пытается сопоставить его с текущими данными;
- при расхождении выводится предупреждение;
- доступны действия `Обновить` и `Сбросить`.

При сохранении состава в команду записываются:

- подразделение;
- ФИО;
- должность;
- роль;
- процент участия.

### 9.3. Шаг 3. Подразделения

Подразделения отображаются плитками.

Требования:

- можно выбрать одно или несколько подразделений;
- есть массовое выделение;
- шаг не переводит пользователя дальше автоматически.

### 9.4. Шаг 4. Сотрудники

Сотрудники отображаются по выбранным подразделениям.

Для сотрудника показываются:

- ФИО;
- должность.

Требования:

- используются плитки с checkbox;
- есть общее массовое выделение;
- внутри подразделения доступен выбор всех сотрудников.

### 9.5. Шаг 5. Период

Период выбирается по годам и месяцам.

Требования:

- месяцы отображаются карточками;
- у месяца отображается число рабочих дней;
- выбор месяцев возможен только подряд;
- для шаблонного экспорта доступно не более трех месяцев.

### 9.6. Шаг 6. Анализ

Анализ выполняется только по:

- выбранной команде;
- выбранным подразделениям;
- выбранным сотрудникам;
- выбранным месяцам.

На шаге отображаются:

- период;
- количество сотрудников;
- количество недель;
- таблица ролей;
- таблица спринтов.

Требования:

- часы учитывают рабочие дни;
- часы учитывают отпуска;
- часы учитывают процент участия сотрудника;
- режим отчета зависит от настройки в разделе `Разное`.

### 9.7. Шаг 7. Ресурсный план

Система формирует Excel-файл по шаблону и выдает его пользователю на скачивание.

## 10. Слайдер настроек

### 10.1. Общие требования

Слайдер:

- открывается справа;
- имеет собственную вертикальную прокрутку;
- содержит кнопки `Закрыть` и `Сохранить`;
- не закрывается автоматически после сохранения;
- выводит статусные сообщения.

Фактическая ширина интерфейса:

- до `65vw`;
- максимум `936px`;
- на мобильной ширине занимает экран целиком.

### 10.2. Верхний уровень вкладок

Верхний уровень содержит вкладки:

- `Роли`
- `Трудовой календарь`
- `Разное`

Все вкладки оформлены в едином underline-стиле.

## 11. Вкладка «Роли»

### 11.1. Общая структура

Раздел `Роли` состоит из вкладок команд.

Требования:

- каждая команда — отдельная вкладка;
- названия вкладок команд отображаются без префиксных цифр;
- вкладки команд можно переставлять drag-and-drop;
- доступны действия:
  - копирование команды;
  - добавление команды.

Внутри выбранной команды доступны вложенные вкладки:

- `Роли`
- `Время`
- `Состав`

Они оформлены в том же стиле, что и верхний уровень вкладок настроек.

### 11.2. Команда

Команда имеет:

- название;
- группы первого уровня;
- собственное распределение времени;
- собственный состав команды.

### 11.3. Вкладка «Время»

Для каждой команды данные независимы.

Содержимое:

1. Таблица распределения времени.
2. Таблица часов по ролям.

В таблице распределения используются группы:

- ключевые задачи;
- поддержка;
- архитектура;
- прочие задачи;
- задачи бизнеса;
- внутренние задачи.

Правила:

- все поля распределения задаются в процентах;
- общий процент отображается справа от заголовка `Распределение времени`;
- отдельный блок `Всего` не используется;
- при отклонении от `100%` итоговый процент подсвечивается как ошибка;
- таблица часов по ролям показывает только роли текущей команды.

### 11.4. Вкладка «Роли»

Внутри команды настраиваются:

1. Группы первого уровня
2. Должности
3. Виды работ

Фактические подписи уровней в интерфейсе:

- первый уровень: `Роль: <название>`;
- второй уровень: `Должность: <название>`;
- третий уровень: `Вид работы: <название>`.

### 11.5. Группы первого уровня

Примеры:

- Аналитики;
- Разработка;
- Тестирование.

Для группы доступны действия:

- удалить;
- поднять вверх;
- опустить вниз;
- перенести в другую команду;
- добавить должность.

### 11.6. Должности

Должность оформляется как аккордеон.

Поля должности:

- название;
- часов в спринте.

`Часов в спринте` вычисляется автоматически как сумма часов всех видов работ.

### 11.7. Виды работ

Вид работы оформляется как вложенный аккордеон.

Поля:

- название вида работы;
- время, час;
- признак `Основная деятельность`;
- описание.

Правила:

- время допускает дробное значение;
- в пределах одной должности основной может быть только один вид работы;
- значение времени участвует в расчете часов должности.

### 11.8. Вкладка «Состав»

Вкладка содержит таблицу сохраненного состава команды.

Колонки:

- Подразделение
- ФИО
- Должность
- Роль
- % участия

Особенности:

- таблица пересобирается при сохранении состава команды;
- колонка `Роль` по умолчанию заполняется значением из `Должность`;
- `Роль` доступна для ручного редактирования;
- `% участия` доступен для ручного редактирования;
- `% участия` — целое число от 0 до 100;
- таблица поддерживает сортировку по всем колонкам;
- у каждого заголовка есть кнопки `↑` и `↓` для направления сортировки;
- заголовки таблицы выровнены по центру;
- данные таблицы выровнены по левому краю;
- поле `Роль` может быть многострочным и подстраивается под высоту строки.

Над таблицей располагаются:

- поясняющий текст;
- кнопка `Сброс` по правому краю.

## 12. Вкладка «Трудовой календарь»

Содержит форму подключения внешнего календарного API.

Требования:

- кнопка `Тест` находится справа в верхней части панели;
- по результату проверки выводится информационное окно;
- доступно закрытие окна кнопкой `Закрыть`.

## 13. Вкладка «Разное»

Содержит только режим формирования отчета.

Доступные варианты:

- `По ролям`
- `По должностям`

Логика:

- `По ролям` — отчет агрегируется по группам первого уровня;
- `По должностям` — отчет строится по отдельным должностям.

Поля `Длительность спринта` и `Начало спринта` в пользовательском интерфейсе не отображаются, но поддерживаются в backend-модели настроек для расчетной логики.

## 14. Расчетная логика

### 14.1. Базовые правила

Расчет строится по рабочим дням периода и учитывает:

- производственный календарь;
- отпуска сотрудников;
- процент участия сотрудника;
- выбранный период;
- режим формирования отчета.

### 14.2. Роли и должности

Роль сотрудника определяется:

- по должности из файла отпусков;
- либо по пользовательскому назначению роли, если это предусмотрено настройками.

### 14.3. Спринты

План разбивается по спринтам, а каждый спринт содержит:

- номер;
- диапазон дат;
- рабочие дни.

## 15. API

Приложение использует следующие backend-методы:

- `GET /api/bootstrap`
- `POST /api/upload-vacations`
- `PUT /api/settings`
- `POST /api/reset-workflow`
- `GET /api/calendar/month-workdays`
- `POST /api/analyze`
- `POST /api/team-members/save`
- `POST /api/team-members/clear`
- `POST /api/team-members/refresh`
- `POST /api/export`

## 16. Выходной файл

Результат экспорта — Excel-файл ресурсного плана, сформированный на основе шаблона `Ресурсный план.xlsx`.

Файл включает данные сводного и детального планирования, а также листы, используемые для расчетов и типовых задач.

## 17. Тестирование

Проект содержит автотесты на:

- парсинг входного Excel;
- работу календарного API и fallback-логики;
- построение ресурсного плана;
- расчеты часов, спринтов и режимов группировки.

Тестовый запуск выполняется командой:

```bash
node --test
```

## 18. Ограничения

- база данных не используется;
- данные хранятся в файловой системе;
- корректность расчета зависит от структуры входного Excel-файла;
- корректность рабочих дней зависит от календарного API либо fallback-логики;
- экспорт ограничен шаблоном ресурсного плана и не поддерживает произвольные форматы выгрузки.
